# CLAUDE.md

Guidelines for Claude Code (claude.ai/code) when working in this repository.

## Project Overview

WebFly is a Flutter-based hybrid runtime that renders React web apps inside a native shell via [WebF](https://github.com/openwebf/webf). The Flutter host provides native capabilities (BLE, permissions, SQLite, sharing) exposed to the web layer through a module invocation bridge.

## Monorepo Structure

- **`lib/`** — Flutter/Dart host app (launcher, routing, asset HTTP server, WebF module registration)
- **`webfly_packages/`** — Feature packages, each containing Dart + TypeScript dual-side code:
  - `webfly_bridge/` — Wire format + `createModuleInvoker` / `WebfModuleEventBus`
  - `webfly_ble/` — BLE module (wraps `flutter_blue_plus`)
  - `webfly_permission/` — Permission module (wraps `permission_handler`)
  - `webfly_theme/` — Theme module
- **`frontend/`** — React + Vite web app (UI running inside WebF)
- **`contrib/webf_usecases/`** — Example use-case apps (React and Vue)
- **`flutter_tools/`** — Rust-based build/release tools, invoked via `just` (git submodule)
- **`platforms/`** — Platform templates (Android manifest, signing config)

## Android Directory

`android/` is generated by `flutter_tools/flutter_gen_platforms`: reads config from `app.pkl` → `flutter create` → copies and template-substitutes from `platforms/android/`. When modifying `android/`, prefer editing `platforms/android/` and `app.pkl` instead. Currently only Android is targeted.

## Build & Development Commands

Prefer commands defined in `justfile`. Run `just --list` to see all available commands.

### Root (requires `just` task runner)

```sh
just setup               # One-command init (pixi + deps + gen + hooks)
just generate            # Dart build_runner (root + webfly_ble)
just android             # Run on Android (debug), long-running, logs to logs/
just android release     # Release mode, long-running, logs to logs/
just build-apk           # Release APK (obfuscated)
just test                # Flutter tests
just test-frontend       # Frontend vitest
just test-all            # All tests
just format              # dart format lib test
just format-check        # CI format gate
just analyze             # flutter analyze
just lint                # format-check + analyze + frontend lint/tsc
just ci                  # Full CI pipeline (setup → lint → test → build)
```

### Frontend (`frontend/`)

```sh
pnpm dev                 # Vite dev server (auto-port 5173+, prints QR code)
pnpm build               # tsc -b && vite build
pnpm lint                # ESLint
pnpm test                # Vitest
pnpm check:webf-css      # WebF CSS constraint check
```

## Architecture

```
Flutter Host  ──WebF Runtime──▶  React App (Vite bundle)
     │                               │
     │  webf.invokeModuleAsync()     │  @webfly/ble, @webfly/permission
     │◀──────────────────────────────│  (TS typed wrappers, neverthrow Result)
     │
     ├─ Asset HTTP Server (shelf, auto-port)
     ├─ GoRouter (native pages + WebF views)
     └─ WebF Modules registered in lib/main.dart
```

- **Native → Web**: Modules defined in Dart (`WebF.defineModule`), called from TS via `webf.invokeModuleAsync(moduleName, method, ...args)`
- **Event bus**: `WebfModuleEventBus` for subscriptions (e.g., BLE scan result streaming)
- **Routing**: GoRouter (Flutter side) manages native/WebF pages; React Router manages web app internal routing. Route focus tracking ensures the correct WebF controller is active
- **State management**: Dart uses `signals_flutter`; web uses `zustand` + `@tanstack/react-query`

## Key Conventions

### TypeScript / Frontend

- **Strict TypeScript** — `strict: true`, `noUnusedLocals`, `noUnusedParameters`, `verbatimModuleSyntax`
- **Path aliases** — `@webfly/ble`, `@webfly/permission`, `@webfly/theme` resolve to `../webfly_packages/*/lib/`
- **Error handling** — `neverthrow` Result types, ESLint enforced (`must-use-result: error`)
- **React Compiler** enabled via `babel-plugin-react-compiler`

### WebF CSS Constraints

WebF only supports a CSS subset; the project enforces checks at build time:
- Tailwind **preflight disabled** (injects unsupported properties)
- **Opacity plugins disabled** (`textOpacity`, `backgroundOpacity`, `borderOpacity`, etc.)
- **Dark mode**: must use `media` strategy (not `class`)
- Run `just webf-check` to verify CSS properties against the WebF whitelist

### Dart / Flutter

- **State management**: `signals` / `signals_flutter` (not Provider/Bloc)
- **Logging**: `talker` (global instance)
- **Code generation**: `build_runner` — run `just generate` after modifying annotated classes (especially `webfly_ble`)

## Git

Run `just format` before committing. Commit messages follow Conventional Commits:

```
<type>(<scope>): <description>

[optional body]
```

Common types: `feat`, `fix`, `docs`, `refactor`, `style`, `test`, `chore`. Messages should include key info about new files, feature changes, etc.

Confirm before running `git tag`, `git push`, `git checkout`, or other commands that modify the working directory or remote state.

## Release Process

When the user says "release", **always use `just release [major|minor|patch]`** (defaults to patch). Do not manually run git tag / push. The command automatically formats, bumps version, commits, tags, and pushes.

Before releasing:

1. Check submodules (`flutter_tools`, `contrib/webf_usecases`, `webfly_packages`) for uncommitted changes; commit & push in each submodule first
2. Update the main repo's submodule references (`git add <submodule_path>`)
3. Review all changes since the last release (`git log <last-tag>..HEAD`), determine if READMEs (EN + CN) need updating (new features, command changes, config changes, etc.); update and commit if needed
4. Commit any remaining main repo changes
5. Run `just release`, `just release minor`, or `just release major`

## LED Effect System

Effects are located at `frontend/public/effects/{effectId}/`, containing:
- `meta.json` — Name, description
- `ui.json` — JSON-render spec (parameter UI) + optional `bridge`/`speed` config
- `effect.ts` — Effect logic (compiled at runtime via Babel `typescript` + `transform-modules-commonjs`)

Runtime code (`effect-runtime.ts`) provides `createBaseMachine`, `makeBlank`, `toRgb`, `hsvToRgb`. Effects export a `createEffect` function returning an `EffectMachine`.

`EffectRenderer` compiles and runs effects; `DeviceCanvasView` renders the device LED layout on canvas at 50ms polling intervals.
